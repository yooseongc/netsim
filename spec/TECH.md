# netsim 기술 스택 요구사항

## 1. 개요

`netsim`은 Linux 네트워크 패킷 처리 경로를 시뮬레이션하는 Web 기반 도구이다.  
배포는 Docker를 기본으로 하며, 하나의 컨테이너 안에서 frontend와 backend를 함께 제공하는 구조를 사용한다.  
설정 데이터와 시나리오 데이터는 DB 없이 파일 기반으로 관리한다.

---

## 2. 기본 기술 방향

- 단일 Docker 컨테이너로 배포할 수 있어야 한다.
- Web UI와 시뮬레이션 엔진은 논리적으로 분리하되, 동일 컨테이너 안에서 함께 동작할 수 있어야 한다.
- 백엔드는 Rust 기반으로 구현해야 한다.
- 프론트엔드는 React + TypeScript 기반으로 구현해야 한다.
- 설정 및 시나리오 데이터는 파일로 저장하고 관리해야 한다.
- Linux 시스템 설정 import를 지원할 수 있어야 한다.
- 정적 링크 및 경량 배포를 위해 Rust `musl` 타깃을 우선 고려해야 한다.

---

## 3. 기술 스택 요구사항

## 3.1 백엔드

### TS-001. 구현 언어
백엔드는 **Rust 1.93**를 사용해야 한다.

### TS-002. 웹 프레임워크
백엔드 API 서버는 **axum** 기반으로 구현해야 한다.

### TS-003. 비동기 런타임
비동기 처리 및 요청 처리를 위해 **tokio**를 사용해야 한다.

### TS-004. 직렬화 및 설정 처리
데이터 직렬화와 설정 파일 처리를 위해 다음을 사용해야 한다.

- `serde`
- `serde_json`
- `serde_yaml`

### TS-005. 시뮬레이션 엔진 구조
시뮬레이션 엔진은 Web 계층과 분리된 내부 모듈 또는 라이브러리 형태로 구성해야 한다.

### TS-006. 시스템 설정 import 방식
Linux 시스템 설정 import는 다음과 같은 외부 명령 실행 결과를 파싱하는 방식으로 구현해야 한다.

- `ip addr`
- `ip rule`
- `ip route`
- `nft list ruleset`
- `iptables-save`

### TS-007. API 형태
백엔드는 HTTP 기반 API를 제공해야 하며, 최소한 다음 기능을 포함해야 한다.

- 시뮬레이션 실행
- 시나리오 저장 및 불러오기
- 시스템 설정 import
- import 결과 검증
- 결과 조회

---

## 3.2 프론트엔드

### TS-010. 구현 언어
프론트엔드는 **TypeScript**를 사용해야 한다.

### TS-011. UI 프레임워크
프론트엔드는 **React** 기반으로 구현해야 한다.

### TS-012. 빌드 도구
프론트엔드 빌드는 **Vite**를 사용해야 한다.

### TS-013. UI 역할
프론트엔드는 최소한 다음 기능을 제공해야 한다.

- 시나리오 입력 및 편집
- 시스템 설정 import 실행
- 시뮬레이션 실행
- trace 결과 표시
- 단계별 상태 변화 시각화
- explain 결과 표시

### TS-014. 백엔드 연동
프론트엔드는 백엔드 API와 HTTP로 통신해야 한다.

### TS-015. 정적 자산 제공
빌드된 프론트엔드 정적 파일은 동일 컨테이너 내에서 백엔드 또는 내장 정적 파일 서버를 통해 제공되어야 한다.

### TS-016. CSS 프레임워크

- shadcn/ui, tailwind 사용을 고려한다.
- 자체 UI 컴포넌트를 정의하고 이를 철저히 사용한다.

---

## 3.3 데이터 및 파일 관리

### TS-020. 저장 방식
시스템은 DB를 사용하지 않고 파일 기반으로 데이터를 관리해야 한다.

### TS-021. 저장 대상
파일로 저장해야 하는 대상은 다음을 포함해야 한다.

- 시나리오 정의
- import된 설정 데이터
- 사용자 편집 결과
- 필요 시 시뮬레이션 결과 스냅샷

### TS-022. 파일 형식
기본 저장 형식은 **YAML**로 하고, 필요 시 **JSON**도 지원할 수 있어야 한다.

### TS-023. 파일 구조
프로젝트 또는 작업 단위로 파일을 구분하여 저장할 수 있어야 한다.

예:
- `projects/<project-name>/scenario.yaml`
- `projects/<project-name>/imported-config.yaml`

### TS-024. 독립 실행성
저장된 파일만으로 동일한 시뮬레이션 환경을 다시 로드하고 재실행할 수 있어야 한다.

---

## 3.4 배포 및 런타임

### TS-030. 배포 단위
배포는 **Docker 이미지**를 기본 단위로 해야 한다.

### TS-031. 단일 컨테이너 구성
frontend와 backend는 **하나의 컨테이너** 안에서 함께 동작해야 한다.

### TS-032. 컨테이너 내부 구성
컨테이너 내부에서는 다음 중 하나의 방식으로 frontend와 backend를 제공할 수 있어야 한다.

- Rust backend가 정적 frontend 파일을 함께 서빙하는 방식
- 별도 내장 웹서버 없이 단일 Rust 프로세스가 API와 정적 파일 제공을 모두 처리하는 방식

### TS-033. 기본 실행 방식
컨테이너 실행 시 하나의 진입 프로세스로 서비스가 시작되어야 한다.

### TS-034. 설정 파일 마운트
사용자는 컨테이너 볼륨 마운트를 통해 시나리오 및 설정 파일을 외부에서 주입하거나 보존할 수 있어야 한다.

### TS-035. 로그 출력
애플리케이션 로그는 컨테이너 표준 출력(stdout/stderr)으로 기록되어야 한다.

---

## 3.5 빌드 전략

### TS-040. Docker 기반 빌드
기본 빌드 방식은 Docker 멀티스테이지 빌드를 사용해야 한다.

### TS-041. Rust 타깃
Rust 바이너리는 기본적으로 **musl 타깃**을 사용하여 빌드할 수 있어야 한다.

예:
- `x86_64-unknown-linux-musl`

### TS-042. 경량 런타임 이미지
최종 런타임 이미지는 가능한 한 경량으로 구성해야 한다.

### TS-043. 프론트엔드 빌드 산출물 포함
프론트엔드 빌드 결과물은 최종 런타임 이미지에 포함되어야 한다.

### TS-044. 단일 이미지 실행 가능성
최종 이미지는 별도 프론트엔드 컨테이너 없이 단독으로 실행 가능해야 한다.

---

## 3.6 운영체제 및 실행 환경

### TS-050. 대상 환경
시스템은 Linux 환경에서 동작해야 한다.

### TS-051. 시스템 설정 import 전제
시스템 설정 import 기능은 컨테이너 내부에서 대상 시스템 명령 실행 결과에 접근할 수 있는 환경을 전제로 한다.

### TS-052. 격리된 시뮬레이션 보장
import된 설정은 실제 시스템 상태와 분리된 내부 데이터로 변환되어 시뮬레이션에 사용되어야 한다.

---

## 3.7 선택 기술

### TS-060. OpenAPI 문서화
필요 시 API 문서화를 위해 OpenAPI 생성 도구를 사용할 수 있어야 한다.

권장 예:
- `utoipa`

### TS-061. 파일 변경 감지
필요 시 설정 파일 자동 반영을 위해 파일 변경 감지 라이브러리를 사용할 수 있어야 한다.

권장 예:
- `notify`

### TS-062. 시각화 라이브러리
프론트엔드 시각화 기능 강화를 위해 별도 라이브러리를 사용할 수 있어야 한다.

예:
- flow diagram용 라이브러리
- 코드/설정 편집기
- diff 뷰어

## 5. 권장 아키텍처

### 구조
- React/Vite로 frontend를 빌드한다.
- Rust/axum backend가 API를 제공한다.
- backend가 frontend의 정적 파일도 함께 제공한다.
- 시나리오와 설정은 컨테이너 내 특정 디렉터리에 파일로 저장한다.
- 운영 시 해당 디렉터리는 볼륨 마운트로 외부와 연결할 수 있어야 한다.

### 예시 디렉터리 구조
- `/app/netsim` : 실행 바이너리
- `/app/ui` : 정적 frontend 파일
- `/data/projects` : 시나리오 및 설정 저장 경로

## 6. MVP 기준 기술 범위

초기 버전은 다음 범위를 포함해야 한다.

- Rust 1.93 기반 backend
- axum 기반 API
- React + TypeScript + Vite 기반 frontend
- Docker 멀티스테이지 빌드
- 단일 컨테이너 배포
- YAML/JSON 파일 기반 시나리오 저장
- 시스템 설정 import
- Web UI를 통한 시뮬레이션 실행 및 trace 조회

---

## 7. 제외 

다음 항목을 필수로 하지 않는다.

- 외부 DB 연동
- frontend/backend 분리 배포
- Kubernetes 배포
- 멀티 컨테이너 오케스트레이션
- 실시간 협업 편집
- 분산 처리 구조
